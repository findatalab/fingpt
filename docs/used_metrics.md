# Используемые метрики для оценки RAG-чатбота (Retrieval + Generation)
Типы используемых метрик:
- `ir_metrics (information retrieval)` — метрики **информационного поиска** (retrieval): насколько хорошо ретривер находит нужные куски (чанки).
- `LLM-as-a-judge` — метрики, которые оценивают **качество ответа** (через LLM-судью): насколько выбранные куски действительно полезны для ответа и/или насколько ответ опирается на них.

---

## 1) IR-метрики (`ir_metrics`)

### Термины: `top-k`, `gold_chunks_id`
Ретривер возвращает список чанков `retrieved = [c1, c2, c3, ...]` в порядке убывания уверенности/сходства в чанках, которые больше всего подходят для ответа на вопрос.
- `top-k` — первые `k` элементов этого списка.
Для IR-метрик считается, что чанк релевантен, если его `chunk_id` совпадает или содержится в `gold_chunks_id`
- `gold_chunks_id` — это идентификатор (или список идентификаторов) **эталонного релевантного чанка** (gold / ground truth), который *должен* быть найден ретривером для данного запроса. 
То есть это id чанков, которые содержат контекст для ответа на вопрос.

IR-метрики зависят от качества gold-разметки. Если gold_chunks выбран слишком узко/широко, метрики будут искажены.

---

### 1.1) `hit@k`
**Что считается:** нашёлся ли хоть один релевантный чанк в топ-k. Возможные значения: {0, 1} или {"ДА", "НЕТ"}

**Формула:**
- `hit@k = 1`, если любой `gold_id` содержится в `top_k`
- иначе `hit@k = 0`

Это “метрика успеха” ретривера. Для RAG очень важно, чтобы нужный контекст был найден в очень маленьком `k` (1–5), иначе генератору не на что опираться.

---

### 1.2) `precision@k`
**Что считается:** доля релевантных чанков среди топ-k.

**Формула:**
- `precision@k = (# релевантных в top_k) / k`

В RAG это полезно, потому что генерация на нерелевантном контексте часто даёт галлюцинации.

---

### 1.3) `recall@k`
**Что считается:** доля найденных релевантных чанков среди всех релевантных.

**Формула:**
- `recall@k = (# релевантных в top_k) / (# всех релевантных)`

В RAG это критично, потому что если нужный фрагмент не попал в top-k, модель физически не сможет на него опереться.

---

### 1.4) `reciprocal rank (RR)`
**Что считается:** обратный ранг первого релевантного чанка.

**Формула:**
`RR = 1 / rank_первого_релевантного`

Если релевантный найден:
- на 1 месте → RR = 1.0
- на 2 месте → RR = 0.5
- на 5 месте → RR = 0.2

Если релевантного нет → RR = 0.

В RAG это важно, потому что ранние позиции сильнее влияют на генерацию (LLM чаще опирается на первые документы).

---

### 1.5) `MRR` (Mean Reciprocal Rank)
**Что считается:** среднее значение reciprocal rank по всем запросам.

**Формула:**
`MRR = (1 / N) * Σ RR_i`

где:
- N — количество запросов
- RR_i — reciprocal rank для i-го запроса

MRR отражает качество ранжирования в среднем:
насколько высоко система помещает первый релевантный документ.

## 2) LLM-as-a-judge-метрики (`judge_metrics`)
LLM-as-a-judge метрики — это оценка качества retrieval и генерации с помощью другой LLM, 
которая выступает в роли эксперта и проверяет смысл, а не только совпадения идентификаторов.

---

### 2.1) `contextual_precision`
**Что считается:** насколько релевантные чанки ранжированы выше нерелевантных.
LLM анализирует порядок retrieved-документов и оценивает, правильно ли система расставила приоритеты.
LLM сильнее опирается на первые и более “значимые” по позиции документы

Интуитивная идея:
- релевантные чанки должны быть выше
- нерелевантные — ниже
- порядок должен отражать важность

Что измеряет:
- корректность порядка документов
- пригодность контекста для генерации

Contextual Precision отвечает на вопрос “Поданы ли важные чанки в правильном порядке для генерации?”

---

### 2.2) `contextual_recall`
**Что считается:** доля фактов из эталонного ответа, которые можно однозначно вывести из переданных чанков.
LLM выступает судьёй и проверяет, можно ли каждый факт из правильного ответа обосновать найденным контекстом.

Что измеряет:
- полноту retrieval
- достаточность контекста для генерации идеального ответа
- качество embedding-модели (находит ли она всё нужное)

Зачем это в RAG:

Если retrieval не содержит всех нужных фактов, генерация никогда не будет полностью корректной — даже при идеальной LLM.
Contextual Recall отвечает на вопрос “Есть ли в переданном контексте вся информация, необходимая для правильного ответа?”

**Чем отличается от IR-`hit@k`:**
- `hit@k` говорит “нашёлся ли хотя бы один gold-чанк”.
- contextual_recall говорит “хватит ли того, что нашли, чтобы ответить”.
